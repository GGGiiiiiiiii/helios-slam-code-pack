## 0. 一句话总览

```
linorobot2 骨架（EKF + Nav2/slam launch + rviz + maps + Gazebo Classic）
        +
Helios 器官移植（3 个 SDK 桥接节点 + 双雷达 URDF + 全向 nav2 参数）
        =
helios_linorobot_ws（新工作空间；现有 helios_nav_ws 保留作对照）
```

linorobot2 的底层是 micro-ROS + 自制硬件，**与珞石 SDK 不通**；可复用的是工程结构、EKF 链路与 launch 范式。Helios 专属的 SDK 桥接必须自写（已在 `helios_nav` 实现，迁入 `helios_bringup` 即可）。

## 1. 什么是「桥接」

厂家底盘是私有接口（`sr_amr_control`），开源导航栈认 ROS 标准接口。**桥接 = 中间薄适配层，不改 SDK、不改 Nav2，只做话题/坐标系/命令翻译。**

```
厂家 SDK (sr_amr_control)        桥接层 (helios_bringup)              开源导航栈
────────────────────────        ─────────────────────────             ──────────────
front/rear scan            ──▶  laser_preprocess              ──▶ /scan ──▶ slam_toolbox / amcl
system_state (vx,vy,wz)    ──▶  odom_publisher (+ 可选 EKF)   ──▶ /odom + TF ──▶ amcl / Nav2
/cmd_vel (Nav2 输出)        ──▶  cmd_vel_relay                  ──▶ remote_control_cmd_vel ──▶ 底盘
```

### 2.2 按状态归类

- **✅ 已具备**：`/scan`、**原生 `/odom`（3.33Hz）**、**IMU（Orbbec 200Hz）**、`/cmd_vel`、速度反馈、`map→odom`（算法）、`/map`、急停、footprint（STL 实测 0.68×0.64×0.408）。
- **⚠️ 有但打折/缺外参**：雷达外参（STL 实测 xy，z/对角待确认）、IMU 安装外参（缺）、限速（保守估计）、TOF（ROS 接口未知）。
- **❌ 缺失**：轮速编码器（原生 odom 已替代其作用）、TOF ROS2 接口、各传感器精确外参。

**结论**：不仅最小闭环，**原生 odom + IMU 也已到位**，真机可上 EKF 融合。剩余主要是**外参标定类**数据（雷达 z/对角、IMU 外参、TOF 接口）与速度上限。

### 2.3 雷达数据限制（实践必读）

- 无效值 **`101.0`** 必须转 `inf`，否则污染地图与定位。
- SDK 从 SROS `loc_laser_points` 重建 `LaserScan`，**非原始 ring data**；仅 LaserScan，无 PointCloud2。
- 雷达默认 **`lidar:=false`**，需 launch 或调服务开启。
- 约 **5Hz / 1°**，移动建图与避障需实测、必要时降速。

## 6. 坐标系（TF）约定 —— 确认命名

```
map → odom → base_footprint → base_link → {right_front_laser_link, left_behind_laser_link, tof_*, imu_link(Orbbec, 外参待标定)}
```

| TF | 发布者 | 备注 |
|----|--------|------|
| `map → odom` | slam_toolbox / AMCL | 互斥 |
| `odom → base_footprint` | **二选一**：EKF（有 IMU 后）或 `odom_publisher` | **同一 TF 只能一个发布者** |
| `base_footprint → base_link` 及传感器 | `robot_state_publisher`（URDF） | 静态 |

**雷达 frame 以 SDK 真机为准**（《雷达测试》已验证）：

- 前雷达：`right_front_laser_link`
- 后雷达：`left_behind_laser_link`

> ⚠️ 全工程统一 `base_footprint`（桥接节点、slam、amcl、costmap、MPPI）。
>
> **里程计/EKF 切换规则（三选一，TF 唯一发布者）**：
> - 阶段 A（回退，无 EKF）：`odom_publisher` 发 `/odom` + `odom→base_footprint` TF（`use_odom:=true publish_tf=true`）。
> - 阶段 B（推荐真机）：EKF 融合原生 `/sr_amr_control/odom` + IMU `/camera/gyro_accel/sample`，发 `/odometry/filtered` + TF（`use_ekf:=true use_odom:=false`）。
> - 阶段 C（仿真）：Gazebo planar_move 直接发 `/odom` + TF。

# 可选 EKF：Node(package="robot_localization", executable="ekf_node", ...)

`linorobot2_gazebo/launch/gazebo.launch.py` 中 URDF 路径指向 `helios_description`。

## 8. costmap 双雷达策略（确认版补充）

- **slam_toolbox / AMCL**：需要单帧 `/scan` → `laser_preprocess` 合并（或后续换 `2D_Scan_Merger_ROS2`）。
- **Nav2 costmap**：**不必合并**，直接双路 `observation_sources`，减少对角拼接损失：

```yaml
obstacle_layer:
  observation_sources: scan_front scan_rear
  scan_front:
    topic: /sr_amr_control/front/scan
    data_type: LaserScan
    clearing: true
    marking: true
    inf_is_valid: true
  scan_rear:
    topic: /sr_amr_control/rear/scan
    data_type: LaserScan
    clearing: true
    marking: true
    inf_is_valid: true
```

前置：`laser_preprocess` 可对两路各跑过滤节点，或 costmap 侧接受已过滤话题。

## 9. 必改 4 处（最容易漏）

1. **AMCL** `robot_model_type: "nav2_amcl::OmniMotionModel"`
2. **MPPI** `motion_model: "Omni"` + `vy_max > 0`
3. **base_frame** 全工程统一 `base_footprint`
4. **TF 唯一发布者**：`odom_publisher` 与 EKF **不能同时**发 `odom→base_footprint`
