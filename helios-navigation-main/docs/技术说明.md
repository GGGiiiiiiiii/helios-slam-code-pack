# Helios 底盘导航 · 技术说明

> 讲清这套代码「怎么串起来」：包职责、端到端数据流、桥接层、TF 树、launch 拼装、仿真与真机差异。

---

## 1. 系统边界

- **本仓库负责**：珞石 Helios 移动底盘的**开源 2D 导航**（建图、定位、路径规划、避障、到点）。
- **不负责**：厂家 MATRIX 导航界面、机械臂控制、头部 Orbbec 的感知业务。
- **总体思路**：厂家 SDK（`sr_amr_control`）只当**数据/控制通道**，用一层薄「桥接」把私有接口翻译成 ROS 标准接口，喂给标准 Nav2 栈。

## 2. 包职责（`lios/helios_linorobot_ws/src/`）

| 包 | 职责 |
|----|------|
| `helios_bringup` | ★核心：3 个桥接节点 + Nav2/SLAM/EKF 配置 + 真机 launch（bringup/mapping/localization/navigation） |

## 3. 端到端数据流（真机）

```
厂家 SDK (sr_amr_control)          桥接层 (helios_bringup)              开源导航栈
──────────────────────────       ────────────────────────           ─────────────────────
front/rear/scan (101.0占位) ─▶ laser_preprocess (净化/合并/自过滤) ─▶ /scan ─▶ slam_toolbox / amcl / costmap
/sr_amr_control/odom(3.33Hz)┐
/camera/gyro_accel/sample   ├─▶ EKF (robot_localization) ───────────▶ /odometry/filtered + odom→base_footprint TF
  (头部 Orbbec, 200Hz)      ┘        （或回退：odom_publisher 速度积分）
                                                                        │
Nav2 输出 /cmd_vel ◀────────── cmd_vel_relay ◀───────────────────────── nav2 controller (MPPI Omni)
   └─▶ /sr_amr_control/remote_control_cmd_vel（需先开远程控制）
```

## 4. 桥接层三节点

### 4.1 `laser_preprocess.py`
- **输入**：`/sr_amr_control/front/scan`、`/rear/scan`（`LaserScan`）。
- **输出**：`/scan`。
- **做三件事**：
  1. **无效值净化**：`ranges >= 101.0`、越界、非有限 → `inf`（否则被当成 101m 处真障碍污染地图）。
  2. **双雷达合并**（`merge=true`）：各帧过滤后借 TF 转到 `base_footprint`，按角度分 bin 取最近，合成 360° 单帧。
  3. **本体自过滤**：落在车身轮廓矩形内（`self_filter_x/y`）的点丢弃——解决对角双雷达**互扫**产生的车身附近假障碍。
- **关键参数**：`front_topic/rear_topic/output_topic/merge/target_frame/invalid_value/range_min/range_max`。

### 4.2 `odom_publisher.py`
- **定位**：真机首选**原生 `/sr_amr_control/odom` + EKF**；本节点是**回退/仿真**用的速度积分里程计。
- **输入**：`/sr_amr_control/system_state`（车体系 vx/vy/wz）。**不用** `current_pose`（那是 MATRIX 定位）。
- **输出**：`/odom`（+ 可选 `odom→base_footprint` TF）。全向：车体系速度按 yaw 旋转到里程计系再积分。
- **会漂移**，靠激光经 slam/amcl 修正 `map→odom`；`max_dt` 守护断流跳变。
- **`publish_tf`**：接 EKF 时设 false（TF 由 EKF 发，避免双发布者）。

### 4.3 `cmd_vel_relay.py`
- **输入** `/cmd_vel`（Nav2 输出 `Twist`）→ **输出** `/sr_amr_control/remote_control_cmd_vel`。
- 启动时调 `remote_control_enabled`（可选 `remote_control_oba_enabled` 远程避障）开远程控制；服务不可用只告警不崩溃（方便无底盘时跑仿真）。
- 保留 `linear.y` 横移（底盘不支持则自动忽略）；退出按需关远程控制。

> 桥接节点依赖厂家自定义类型 `sr_amr_interfaces`。真机需先 `source` 完整 SDK 再编译/运行；仿真不发 `system_state`，odom 由 Gazebo 直接产生，不强依赖。

## 5. TF 树

```
map → odom → base_footprint → base_link → { right_front_laser_link,
                                            left_behind_laser_link,
                                            tof_front/rear/left/right_link }
```

- `map→odom`：**slam_toolbox（建图）** 或 **AMCL（定位）**，互斥不同时跑。
- `odom→base_footprint`：**三选一，唯一发布者**（见下）。
- `base_footprint→base_link→传感器`：`robot_state_publisher` 从 URDF 发。
- **统一 `base_footprint`** 为 base 帧（曾修过 odom_publisher 发 `odom→base_link` 与 URDF 抢父节点、导致 base_link 双父的 bug）。
- **头部 IMU**：不在简化 URDF 里。真机开 EKF 用 IMU 需补 `base_footprint→…→camera/imu` 外参（接整机 URDF 或发静态 TF）。

---

## 6. 里程计三选一（`odom→base_footprint` 唯一发布者）

| 方案 | 参数 | 发 TF 者 | 用途 |
|------|------|----------|------|
| A 原生 odom + EKF | `use_ekf:=true use_odom:=false` | EKF | **真机推荐** |
| B 速度积分回退 | `use_odom:=true use_ekf:=false`（`publish_tf=true`） | odom_publisher | 无 EKF 时 |
| C 仿真 | 两者都关 | Gazebo planar_move | Gazebo |

## 7. Launch 拼装

- `bringup.launch.py`（真机数据层，不含 SLAM/导航，供复用）
  - 节点：`robot_state_publisher`、`laser_preprocess`、`odom_publisher`(条件)、`cmd_vel_relay`(条件)、`ekf_node`(条件)、`static_tf`(条件，与 URDF 二选一)。
  - 参数矩阵：`merge / use_static_tf / use_robot_state_publisher / relay_cmd_vel / use_odom / use_ekf / use_sim_time`。
- `mapping.launch.py` / `localization.launch.py` / `navigation.launch.py`：真机建图 / 定位 / 导航。
- 仿真（`helios_gazebo`）：`sim.launch.py`（Gazebo+模型+激光）→ `sim_slam.launch.py`（+SLAM）→ `sim_nav.launch.py`（+完整 Nav2，用 `IncludeLaunchDescription` + `TimerAction` 控时序）。
- 仿真里用 `/usr/bin/python3` 显式跑 `spawn_entity.py` 并清理 conda 环境（否则 spawn 失败）。

## 8. Nav2 配置要点（`config/nav2_params.yaml`）

- 全局规划：`nav2_smac_planner/SmacPlanner2D`（曾试 `SmacPlannerLattice`+`ConstrainedSmoother` 求平滑，按要求已还原）。
- 局部控制：**MPPI，`motion_model: Omni`**（支持 vy 横移）。
- 定位：AMCL `OmniMotionModel`。
- footprint：`[[0.34,0.32],[0.34,-0.32],[-0.34,-0.32],[-0.34,0.32]]`（STL 实测半尺寸，不含臂）。
- 速度上限：`vx/vy 0.4、wz 1.0` **保守占位**（厂家无规格，待实测）。
- MPPI 轨迹可视化 `visualize: true` → `/trajectories`；局部规划 `/transformed_global_plan`（仅 navigating 时发布）；全局路径 `/plan`。

## 9. 仿真 vs 真机差异

| 项 | 仿真 | 真机 |
|----|------|------|
| odom 来源 | Gazebo planar_move | `/sr_amr_control/odom` + EKF（或 odom_publisher 回退） |
| 激光 | Gazebo 雷达插件 | SDK front/rear scan（需 `lidar:=true` + `lidar_enabled`） |
| IMU | 无 | 头部 Orbbec `/camera/gyro_accel/sample` |
| cmd_vel | 直接进 planar_move | cmd_vel_relay → remote_control（需开远程控制） |
| 传感器类型依赖 | 不需 `sr_amr_interfaces` | 需 source 完整 SDK v1.3.0 |
| base_link 碰撞 | 矮盒/无（planar 2D） | — |
