# Helios 导航 · 典型问题库

> 通用、可复现的坑。每条按「现象 / 常见原因 / 怎么确认 / 怎么处理」。

---

## 仿真

### 1. 重启仿真后没有 `/scan`、TF 一直等 `... to map`
- **现象**：第二次启动一直刷 `Timed out waiting for transform ... base_footprint to map`，`ros2 topic hz /scan` 无数据，SLAM 不建图。
- **常见原因**：上次点了 Gazebo 窗口的叉，只关了 GUI（gzclient），后台 **gzserver 残留**占端口，新 gzserver 起不来 → 无传感器数据。
- **怎么确认**：`pgrep -af gzserver`。
- **怎么处理**：`pkill -9 gzserver gzclient` 后重启。**关仿真一律用终端 Ctrl+C**，别点窗口叉。（`run_sim_nav.bash` 启动前会自动清理。）

### 2. 启动头 20~40 秒刷 transform timeout —— 正常还是卡死？
- **正常**：刷 `... to odom`，是机器人 spawn + planar_move 还没发 odom，会自愈。
- **异常**：刷 `... to map` 且 `/scan` 无数据 → 见问题 1。

### 3. 以为 Nav2 没就绪（`Managed nodes are active` 看不见）
- **现象**：没看到就绪日志。
- **常见原因**：其实已出现，被 slam 的 `Message Filter dropping message ... queue is full`（INFO，非错误）刷屏顶上去了。
- **怎么确认**：另开终端 `ros2 lifecycle get /bt_navigator` → `active [3]` 即就绪。
- **说明**：`dropping message` 是 slam 丢旧激光帧，仿真常见，不影响建图导航。

### 4. `controller_server` 激活超时导致 bringup abort（偶发）
- **现象**：`Failed to change state for node: controller_server`。
- **常见原因**：启动时序竞争 + 机器负载高，服务响应超时。
- **怎么处理**：Ctrl+C 重启一次；资源紧张时多等一会再发目标。

### 5. Gazebo 里「看不到车」但导航能到点
- **现象**：spawn 成功、Nav2 `Goal succeeded`，但 GUI 里无机器人。
- **常见原因**：mesh 用了 `package://...`——**gzserver 能解析，gzclient(GUI) 不能**，visual 加载失败；collision 是 box，物理/导航正常。
- **怎么处理**：URDF mesh 用 `file://$(find helios_description)/...` 绝对路径（server/client/RViz 都认）。symlink-install 下改 xacro 即生效，重启仿真即可。

## RViz / Nav2 使用

### 6. 发目标后车不动 / 狂刷 `Extrapolation Error`
- **常见原因**：① Nav2 未完全 active、TF 未稳就发了目标；② 误用 **2D Pose Estimate**（设初始位姿）当成发目标。
- **怎么处理**：等全部 active 后，用 **Nav2 Goal / 2D Goal Pose** 发新目标，旧的卡死目标会被覆盖。

### 7. `/transformed_global_plan`、`/trajectories` 没有数据
- **常见原因**：这些局部规划/轨迹话题**只在导航进行中（navigating）发布**，目标到达（`Goal succeeded`）后就停。
- **怎么确认**：发一个目标，导航途中再 `ros2 topic hz`。全局路径 `/plan` 则一直有。

### 8. RViz 看不到激光
- **常见原因**：**QoS 不匹配**——激光是 Best Effort，RViz 默认 Reliable；或 Fixed Frame / LaserScan 显示项没配对。
- **怎么处理**：LaserScan 的 Reliability 设 Best Effort；Fixed Frame 用 `map`（导航）或 `odom`（纯仿真）。

## 桥接 / TF（仿真与真机通用）

### 9. 车原地打转不平移，车身四周一圈假障碍
- **常见原因**：对角双雷达都 360°、同高，**互相扫到对方**，合并后假点落在车身附近，costmap 膨胀把车圈死。
- **怎么处理**：`laser_preprocess` 合并环节开**本体自过滤**（`self_filter_x/y` 按真实 footprint 设），丢弃落在车身轮廓内的点。不要用 min-range（互扫点约 0.82m 不近，会误伤近处真障碍）。

### 10. `odom→base_footprint` TF 冲突 / base_link 双父节点
- **常见原因**：`odom_publisher` 与 EKF 同时发 TF；或 odom_publisher 发 `odom→base_link` 与 URDF 的 `base_footprint→base_link` 抢父节点。
- **怎么处理**：里程计**三选一**（见技术说明 §6），保证 `odom→base_footprint` 唯一发布者；开 EKF 时 `use_odom:=false`。全链路统一 `base_footprint`。

### 11. Nav2 插件 fatal：`... does not exist`
- **常见原因**：`nav2_params.yaml` 插件名用了 `::`，Humble 注册名是 `/`。
- **怎么处理**：统一改 `/` 形式，如 `nav2_smac_planner/SmacPlanner2D`、`nav2_bt_navigator/NavigateToPoseNavigator`、`nav2_behaviors/...`。

### 12. 双雷达合并/自过滤失效
- **常见原因**：真机雷达 `frame_id` 与 URDF 里 `right_front_laser_link` / `left_behind_laser_link` 不一致，TF 变换失败。
- **怎么确认**：`ros2 topic echo /sr_amr_control/front/scan --once`（看 `header.frame_id`）。
- **怎么处理**：对齐 URDF frame 名或 laser_preprocess 的 `target_frame`。

## 真机联调

### 13. 真机 `colcon build` 找不到 `sr_amr_interfaces`
- **常见原因**：桥接节点依赖厂家自定义类型，未 source 完整 SDK。
- **怎么处理**：先 `source .../standard_robots_amr_ros2-v1.3.0/install/setup.bash` 再编译/运行。注意 v1.0.5/v1.2.0 **不含雷达模块**，用 Orin 上 v1.3.0。

### 14. 真机底盘不响应 `/cmd_vel`
- **常见原因**：没开远程控制，指令被丢。
- **怎么处理**：确认 `cmd_vel_relay` 已调 `remote_control_enabled`；底盘需手动模式/使能。
